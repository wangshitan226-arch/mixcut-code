# MixCut 上云架构改造方案

> 解决2核4G服务器性能瓶颈，实现可扩展的视频混剪服务
> 
> 文档版本: 1.0
> 更新日期: 2026-04-24

---

## 一、现状分析

### 1.1 当前架构瓶颈

```
┌─────────────────────────────────────────────────────────┐
│  用户浏览器                                               │
│  • 上传素材 → 你的服务器                                  │
│  • 点击播放 → 你的服务器拼接 → 你的服务器传输              │
│  • 文件下载 → 你的服务器                                  │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│  你的服务器 (2核4G)                                       │
│  • 接收上传 → 本地ffmpeg转码                              │
│  • 实时拼接 → 本地ffmpeg concat                          │
│  • 视频传输 → 占用带宽                                    │
│  • 异步上传OSS → 后台任务                                 │
└─────────────────────────────────────────────────────────┘
                           ↓
                    阿里云OSS（存储）
```

**问题：**
- 2核4G既要转码又要拼接又要传输，单用户就占满资源
- 3个用户同时点击预览，服务器直接卡死
- 带宽成为瓶颈，视频传输慢

### 1.2 你的核心诉求

1. **秒级拼接必须保留**（用户体验核心）
2. **预览要流畅**（当前服务器传输慢）
3. **能支撑多用户**（不能3人就卡死）
4. **成本可控**（个人项目，不能太贵）

## 二、上云架构方案

### 2.1 目标架构

```
┌─────────────────────────────────────────────────────────────┐
│  用户浏览器                                                   │
│  • 上传素材 → 直传OSS（绕过你的服务器）                        │
│  • 点击播放 → CDN加速URL（流畅播放）                          │
│  • 本地缓存 → IndexedDB（二次播放秒开）                       │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  你的服务器（只处理API和调度）                                 │
│  • 接收请求 → 调用阿里云ICE/函数计算                          │
│  • 状态管理 → 轮询任务状态                                    │
│  • 元数据存储 → 数据库                                        │
│  • 不处理视频！不传输视频！                                   │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  阿里云服务体系                                               │
│  • OSS（对象存储）← 素材存储、成品存储                        │
│  • CDN（内容分发）← 视频播放加速                              │
│  • ICE（智能媒体）← 视频转码/拼接（可选）                      │
│  • 函数计算FC ← 轻量级视频处理                                 │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 关键改变

| 环节 | 现在 | 上云后 |
|------|------|--------|
| 素材上传 | 传到你的服务器 | **直传OSS**（浏览器→OSS） |
| 素材转码 | 你的服务器ffmpeg | **函数计算FC** 或 **ICE** |
| 视频拼接 | 你的服务器ffmpeg | **函数计算FC** 或保持本地 |
| 视频播放 | 你的服务器传输 | **CDN加速**（OSS+CDN） |
| 视频下载 | 你的服务器传输 | **CDN加速** |
| 你的服务器负载 | 计算+带宽 | **仅API调度** |

## 三、详细改造方案

### 3.1 方案A：保守上云（推荐）

**核心思路：** 保留秒级拼接，其他环节上云

```
素材上传 → 直传OSS → 函数计算转码 → 保存回OSS
                                    ↓
用户点击播放 → 你的服务器秒级拼接（本地copy）→ 返回CDN URL
                                    ↓
                              异步上传成品到OSS
```

**改造点：**

1. **素材上传直传OSS**
   ```javascript
   // 前端：获取OSS临时签名，直传
   const signature = await fetch('/api/oss/signature').then(r => r.json());
   await uploadToOSS(file, signature);
   ```

2. **素材转码用函数计算**
   ```python
   # 上传完成后触发函数计算转码
   # 函数计算：ffmpeg转码 → 保存到OSS
   # 你的服务器只轮询状态
   ```

3. **播放用CDN加速**
   ```python
   # 返回OSS CDN URL，不经过你的服务器
   video_url = f"https://cdn.yourdomain.com/{oss_key}"
   ```

**优点：**
- 保留秒级拼接（用户体验不变）
- 播放走CDN，流畅
- 你的服务器压力大幅降低

**缺点：**
- 还是需要本地拼接（不能彻底解决并发问题）

**成本预估：**
- OSS存储：~¥10/月（100GB）
- CDN流量：~¥20/月（100GB流量）
- 函数计算：~¥5/月（转码调用）
- **总计：~¥35/月**

---

### 3.2 方案B：完全上云（推荐度次之）

**核心思路：** 拼接也上云，你的服务器只做API

```
用户点击播放 → 你的服务器提交ICE任务 → 阿里云ICE拼接
                                    ↓
                              ICE回调通知完成
                                    ↓
                         返回OSS CDN URL给用户
```

**改造点：**

1. **使用阿里云ICE进行拼接**
   ```python
   # 提交拼接任务到ICE
   job_id = ice_client.submit_concat_job(
       inputs=[oss_url1, oss_url2, ...],
       output=f"oss://bucket/renders/{combo_id}.mp4"
   )
   ```

2. **轮询任务状态**
   ```python
   # 前端轮询 /api/tasks/{job_id}/status
   # 你的服务器查询ICE状态返回
   ```

3. **播放走CDN**
   ```python
   # ICE完成后，返回CDN URL
   video_url = f"https://cdn.yourdomain.com/renders/{combo_id}.mp4"
   ```

**优点：**
- 你的服务器零负载（纯API）
- 可支撑无限并发
- 专业视频处理，稳定性高

**缺点：**
- 拼接时间：3-10秒（ICE排队+处理）
- 成本较高

**成本预估：**
- OSS存储：~¥10/月
- CDN流量：~¥20/月
- ICE处理：~¥50-100/月（按调用次数）
- **总计：~¥80-130/月**

---

### 3.3 方案C：混合架构（最推荐）

**核心思路：** 预览用本地秒级，下载用云端处理

```
预览流程（快）：
用户点击播放 → 本地秒级拼接 → 返回CDN URL（低码率预览版）
                                    ↓
                              异步ICE生成高清版

下载流程（慢但质量好）：
用户点击下载 → 等待ICE完成 → 返回高清CDN URL
```

**改造点：**

1. **预览版本地秒级拼接（低码率）**
   ```python
   # 本地拼接 720p/1.5M 预览版
   fast_concat_videos(inputs, preview_path, quality='low')
   # 上传预览版到OSS，返回CDN URL
   ```

2. **高清版异步ICE生成**
   ```python
   # 同时提交ICE任务生成1080p高清版
   ice_job_id = submit_ice_concat(inputs, quality='high')
   # 存到数据库，下载时用
   ```

3. **播放预览版，下载高清版**
   ```python
   # 播放：返回预览版CDN URL（文件小，加载快）
   # 下载：等待高清版完成，返回高清CDN URL
   ```

**优点：**
- 预览秒开（用户体验好）
- 下载质量高（专业处理）
- 服务器负载适中

**缺点：**
- 实现复杂度高
- 存储双份（预览+高清）

**成本预估：**
- OSS存储：~¥20/月（双份）
- CDN流量：~¥30/月
- ICE处理：~¥30/月（仅下载时触发）
- **总计：~¥80/月**

| 3-4 | 上传回调处理 | `backend/routes/upload.py` |

**关键代码：**
```python
# backend/routes/oss.py
@oss_bp.route('/upload/signature', methods=['POST'])
def get_upload_signature():
    """获取OSS临时上传签名"""
    filename = request.json.get('filename')
    
    # 生成临时签名URL
    auth = oss2.Auth(access_key_id, access_key_secret)
    bucket = oss2.Bucket(auth, endpoint, bucket_name)
    
    key = f"uploads/{user_id}/{uuid.uuid4().hex}_{filename}"
    url = bucket.sign_url('PUT', key, 3600)  # 1小时有效期
    
    return jsonify({
        'url': url,
        'key': key,
        'cdn_url': f"https://cdn.yourdomain.com/{key}"
    })
```

# backend/routes/renders.py
def get_video_url(render):
    """返回CDN加速URL"""
    if render.oss_url:

        return render.oss_url.replace(
            'fengma-materials.oss-cn-beijing.aliyuncs.com',

## 六、关键决策

### 6.1 要不要把拼接也上云？

| 场景 | 建议 |
|------|------|
| 个人项目，预算有限 | **不上云**，保留本地秒级拼接 |
| 用户增长快，并发高 | **上云**，用ICE处理 |
| 追求极致体验 | **混合方案**，预览本地+下载云端 |

### 6.2 CDN必须上吗？

**必须上！** 这是解决播放卡顿的关键。

- 现在：你的服务器带宽 1-5MB/s
- CDN：全国节点，用户就近访问 10-50MB/s
- 成本：¥20/月，性价比极高

### 6.3 2核4G服务器还要吗？

**要！** 但角色变了：

- 现在：计算+存储+带宽
- 上云后：纯API调度+数据库
- 2核4G足够支撑100+并发API请求

## 八、总结

**核心结论：**

1. **CDN必须上**（解决播放卡顿，¥20/月）
2. **素材上传直传OSS**（释放服务器带宽）
3. **秒级拼接可以保留**（本地处理，用户体验好）
4. **转码可以上云**（函数计算，按量付费）
5. **拼接上云看预算**（ICE成本高，但可扩展）

**推荐路径：**

```
现在（卡）
  ↓ 第1周
配置CDN（¥20/月，播放流畅）
  ↓ 第2-3周
素材直传OSS（服务器压力降80%）
  ↓ 第4周起（可选）
转码上云 → 拼接上云（完全serverless）
```

**预期效果：**
- 播放流畅度：从卡顿 → 流畅
- 并发能力：从1-2人 → 10-20人
- 服务器负载：从100% → 20%
- 月成本：从¥50 → ¥35-80
