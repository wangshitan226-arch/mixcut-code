# MixCut 存储架构分析：本地存储 vs 阿里云OSS

## 一、当前存储模式分析

### 1.1 当前架构

```
当前存储流程
├── 用户上传视频 → backend/uploads/
├── 转码统一格式 → backend/unified/
├── 混剪结果 → backend/renders/
└── 语音识别 → 需要视频URL（目前用本地路径或临时URL）
```

### 1.2 当前模式的隐患

| 隐患 | 说明 | 风险等级 |
|------|------|----------|
| **服务器磁盘限制** | 视频文件大，服务器磁盘容易满 | 🔴 高 |
| **无持久化保障** | 服务器重启/故障，文件丢失 | 🔴 高 |
| **无法水平扩展** | 多台服务器无法共享文件 | 🔴 高 |
| **语音识别受限** | DashScope需要公网URL，本地文件需临时方案 | 🟡 中 |
| **带宽瓶颈** | 视频下载占用服务器带宽 | 🟡 中 |
| **备份困难** | 需要自行实现备份策略 | 🟡 中 |

---

## 二、阿里云OSS方案分析

### 2.1 推荐架构

```
OSS存储架构
├── 用户上传视频 → 直传OSS (bucket/uploads/)
├── 转码统一格式 → OSS (bucket/unified/) 或 本地处理后上传
├── 混剪结果 → OSS (bucket/renders/)
├── 语音识别 → 直接使用OSS URL
└── 数据库 → 只存储OSS URL，不存本地路径
```

### 2.2 OSS方案优势

| 优势 | 说明 |
|------|------|
| **无限存储空间** | 按需扩展，不受服务器磁盘限制 |
| **高可靠性** | 99.999999999%数据可靠性，多副本存储 |
| **CDN加速** | 全球CDN节点，用户下载速度快 |
| **直传支持** | 客户端直传OSS，不经过服务器 |
| **与阿里云服务集成** | FunASR、视频处理等直接读取OSS |
| **成本可控** | 按量付费，存储+流量分开计费 |
| **自动备份** | 版本控制、跨区域复制等 |

### 2.3 OSS方案劣势

| 劣势 | 说明 |
|------|------|
| **成本增加** | 存储费 + 流量费 + 请求费 |
| **架构复杂度** | 需要处理STS临时凭证、回调等 |
| **网络依赖** | 强依赖阿里云网络稳定性 |
| **数据迁移成本** | 已有数据需要迁移到OSS |

## 三、成本对比分析

### 3.1 假设场景

- 月活跃用户：1000人
- 人均上传视频：10个
- 平均视频大小：50MB
- 月新增存储：1000 × 10 × 50MB = 500GB
- 月视频播放次数：10000次

### 3.2 本地存储成本

| 项目 | 成本 |
|------|------|
| 服务器磁盘（1TB SSD） | ¥200-300/月 |
| 服务器带宽（100Mbps） | ¥500-800/月 |
| 备份存储 | ¥100/月 |
| **总计** | **约 ¥800-1200/月** |

### 3.3 阿里云OSS成本

| 项目 | 单价 | 月费用 |
|------|------|--------|
| 标准存储（500GB） | ¥0.12/GB/月 | ¥60 |
| 外网流出流量（假设500GB） | ¥0.8/GB | ¥400 |
| PUT请求（1万次） | ¥0.01/千次 | ¥0.1 |
| GET请求（10万次） | ¥0.01/千次 | ¥1 |
| CDN流量（可选，500GB） | ¥0.24/GB | ¥120 |
| **总计（无CDN）** | | **约 ¥460/月** |
| **总计（有CDN）** | | **约 ¥580/月** |

**结论**：OSS方案成本更低，且可靠性更高。

## 四、分阶段迁移方案

### 阶段一：混合模式（推荐立即实施）

**目标**：解决语音识别问题，最小改动

```
架构
├── 用户上传 → 本地uploads/（保持现有）
├── 转码 → 本地unified/（保持现有）
├── 混剪结果 → 本地renders/（保持现有）
└── 语音识别前 → 自动上传到OSS临时bucket，获取URL
```

**改动点**：
1. 创建OSS bucket用于临时存储
2. 语音识别前，自动将视频上传到OSS
3. 识别完成后，删除OSS临时文件（或设置生命周期自动删除）

**优点**：
- 改动最小，快速解决语音识别问题
- 现有流程不受影响

**缺点**：
- 需要上传两次（本地+OSS）
- 临时文件管理复杂

### 阶段二：新文件走OSS（推荐3个月内实施）

**目标**：新上传的文件直接走OSS

```
架构
├── 用户上传 → 客户端直传OSS
├── 数据库 → 存储OSS URL
├── 转码 → 从OSS下载 → 处理 → 上传回OSS
├── 混剪结果 → 直接保存到OSS
└── 语音识别 → 直接使用OSS URL
```

**改动点**：
1. 前端实现OSS直传（使用STS临时凭证）
2. 后端生成STS Token
3. 修改Material模型，支持OSS URL
4. 转码服务改为从OSS读取/写入

**优点**：
- 服务器无存储压力
- 支持水平扩展
- 语音识别无阻碍

**缺点**：
- 改动较大，需要前后端配合

### 阶段三：完全迁移（推荐6个月内实施）

**目标**：所有文件都迁移到OSS

**方案**：
1. 编写数据迁移脚本，将现有文件上传到OSS
2. 更新数据库，替换file_path为OSS URL
3. 下线本地存储逻辑

## 五、针对开拍功能的建议

### 当前问题

开拍功能（文字快剪）需要：
1. 视频URL用于语音识别（DashScope要求公网URL）
2. 视频URL用于前端播放
3. 渲染后的视频存储

如果继续使用本地存储：
- 需要临时上传到OSS获取URL用于识别
- 或者使用内网穿透工具（如ngrok），不稳定

### 推荐方案

**短期（立即实施）**：
```python
# 语音识别前，自动上传到OSS
@kaipai_bp.route('/kaipai/<edit_id>/transcribe', methods=['POST'])
def start_transcription(edit_id):
    edit = KaipaiEdit.query.get(edit_id)
    render = Render.query.get(edit.render_id)
    
    # 本地文件路径
    local_path = render.file_path
    
    # 上传到OSS获取临时URL
    oss_key = f"temp/asr/{edit_id}/{os.path.basename(local_path)}"
    upload_to_oss(local_path, oss_key)
    
    # 获取签名URL（有效期1小时）
    video_url = get_oss_signed_url(oss_key, expiration=3600)
    
    # 启动ASR任务
    create_asr_task(edit_id, video_url)
    
    # 设置OSS文件生命周期，1天后自动删除
    set_oss_lifecycle(oss_key, days=1)
```

**长期（跟随阶段二）**：
- 混剪结果直接保存到OSS
- 开拍功能直接使用OSS URL

## 六、最终建议

### 推荐路径

```
现在（立即）
    ↓
实施阶段一：混合模式
- 保持现有本地存储
- 语音识别前自动上传到OSS临时bucket
- 解决开拍功能的URL问题

3个月内
    ↓
实施阶段二：新文件走OSS
- 新上传的文件直传OSS
- 逐步减少本地存储依赖

6个月内
    ↓
实施阶段三：完全迁移
- 历史数据迁移到OSS
- 下线本地存储
```

### 如果不迁移到OSS的替代方案

如果坚持本地存储，解决语音识别问题的方法：

**方案A：内网穿透（不推荐生产环境）**
- 使用 ngrok / frp 等工具暴露本地文件
- 不稳定，不适合生产

**方案B：本地ASR服务（成本高）**
- 部署本地FunASR服务
- 需要GPU服务器，成本高

**方案C：文件中转服务（复杂）**
- 用户上传文件到临时中转服务
- 复杂，不如直接用OSS

### 结论

**强烈建议迁移到OSS**，原因：
1. 成本更低（¥460 vs ¥1000/月）
2. 可靠性更高（99.999999999% vs 自行维护）
3. 与阿里云服务集成更好（语音识别、视频处理）
4. 支持业务扩展（水平扩展、CDN加速）

短期可以先实施"混合模式"解决开拍功能的迫切需求，长期逐步完全迁移。
